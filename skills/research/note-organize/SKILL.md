## Why This Skill Exists

笔记整理是一项多步骤的协调任务：先搜索发现、再分析现状、然后规划方案、接着逐步执行（标准化、建链接、创建索引），最后验证质量。每个步骤都有对应的专业技能，但用户不想逐个调用——他们只想说"帮我整理这些笔记"。

这就是 note-organize 的角色：**编排器**。它负责规划方案、委派执行、验证结果，而不是自己实现每个步骤。

独特能力（无其他技能覆盖）由本技能自己实现：
- **合并重复笔记**（相似度 ≥ 85% 才建议合并）
- **目录结构调整**（移动文件、重组目录）
- **论文目录整理专项**（批量论文笔记的分类、概览、索引）

## Critical Rules

| Rule | Description |
|------|-------------|
| **委派优先** | 不自己实现其他技能的能力。标准化交给 `/note-standardize`，链接交给 `/note-link`，搜索交给 `/obsidian-search`，分析交给 `/note-analyze`，索引交给 `/note-template` |
| **obsidian-search 优先** | 所有搜索/发现操作先用 `/obsidian-search`。它提供五种搜索引擎（关键词搜索、结构化查询、链接关系、语义搜索、标签统计），远比 Glob 强大。仅当 obsidian-cli 不可用时才降级到 Glob/Grep |
| **保留独特能力** | 合并重复、目录整理、论文目录整理专项——这三个能力没有其他技能覆盖，必须自己实现。操作时参考 `ref/obsidian-markdown.md` 获取格式规范 |
| **分步确认** | 每步委派前先规划，委派后验证结果。不做 blind batch。使用 `AskUserQuestion` 展示方案并请用户确认 |
| **质量检查** | 每步完成后验证结果（用 obsidian-search + note-analyze 复查），发现问题则重新委派 |

## Workflow: 10-Step Orchestrator

```
Step 1: 搜索发现 → Step 2: 分析现状 → Step 3: 生成方案 → Step 4: 用户确认
→ Step 5: 标准化 → Step 6: 建立链接 → Step 7: 创建索引
→ Step 8: 合并重复 → Step 9: 目录整理 → Step 10: 质量检查
```

每步完成后记录结果，用于后续步骤的输入和最终质量检查。

### Step 1: 搜索发现

**委派** `/obsidian-search`，根据目标选择搜索模式：

- `obsidian search`: 按关键词快速定位
- `obsidian eval`: 结构化查询（按标签、日期、属性过滤）
- `obsidian backlinks` + `obsidian outgoing`: 获取现有链接关系
- `obsidian tags sort=count counts`: 标签分布统计

搜索范围限制在目标目录（如 `科研/CV`），避免全库搜索。

### Step 2: 分析现状

**委派** `/note-analyze`，重点关注：

- 主题聚类（相似内容发现）
- 孤立笔记检测
- 内容重复分析
- 健康度评估（结构完整性、标签使用情况）
- 改进建议

### Step 3: 生成整理方案

**自己执行**——基于 Step 1 和 Step 2 的输出，规划整理方案：

- 列出要执行的操作及顺序
- 每步的委派目标和预期输出
- 论文目录整理专项的识别（如果目标目录包含大量论文笔记）
- **不要执行任何修改，只做规划**

### Step 4: 用户确认

使用 `AskUserQuestion` 展示方案，让用户选择：

1. 全部执行
2. 选择性执行（可跳过某些步骤）
3. 仅预览（dry run，不修改任何文件）
4. 取消

### Step 5: 标准化格式

**委派** `/note-standardize`，执行范围：

- 标签规范化：中文标签 → 英文层级标签，同义词 → 统一术语，补全缺失标签
- Frontmatter 修复：补全缺失字段，修正格式
- Callout 格式化、Wikilink 修复、公式/语法修复

参考 note-standardize 的 `references/tags_database.md` 确保标签一致性。

### Step 6: 建立链接

**委派** `/note-link`，执行范围：

- 语义关联发现：基于内容相似度和标签重叠
- 智能定位：根据关系类型决定链接位置
- 反向链接：补全双向关联
- 链接验证：用 Glob 确认链接目标存在

### Step 7: 创建索引

**委派** `/note-template`（index 类型），为每个主题簇创建索引笔记。

索引笔记使用表格格式，Wikilink 管道符必须转义（`\|`）。

参考 note-template 的索引模板参考文件。

### Step 8: 合并重复

**自己执行**（独特能力）：

1. Read 两个笔记的完整内容
2. 比较核心观点异同（相似度评估）
3. 相似度阈值：≥ 85% 才建议合并
4. 保留策略：选择内容更完整或标题更规范的版本
5. 在保留笔记中添加 `merged_from` 字段记录合并来源
6. **必须用户确认**才能执行合并或删除

### Step 9: 目录整理

**自己执行**（独特能力）：

1. 使用 Glob 扫描目录结构
2. 识别错位的笔记（内容与目录主题不匹配）
3. 生成移动方案（源路径 → 目标路径）
4. **必须用户确认**才能移动文件
5. 移动后验证所有 Wikilink 是否需要更新

### Step 10: 质量检查

**自己执行**——验证每步结果：

- 用 `obsidian search` 验证搜索结果完整性
- 用 `/note-analyze` 的结构分析复查健康度
- 检查所有创建/修改的笔记格式规范
- 检查所有新链接有效
- 检查索引笔记使用表格格式
- 检查合并后的笔记保留完整内容

## 论文目录整理专项

当目标目录满足以下条件时触发此专项流程：
- 用户明确提到"论文目录"、"Paper 文件夹"
- 目录中包含大量论文笔记（> 10 篇）
- 笔记标题为论文名

### 工作流

```
搜索发现(obsidian-search) → 分析(note-analyze) → 分类规划 → 创建索引(note-template)
→ 主题概览 → 建立链接(note-link) → Idea 捕获(idea-capture) → 移动汇总至 Conclusion/
```

### 输出文件

| 输出 | 执行者 | 说明 |
|------|--------|------|
| 汇总索引（`00_论文索引_[主题].md`） | `/note-template` | 按研究方法分类（表格格式），含概率框架分析和交叉关联 |
| 主题概览（`主题_[名称].md`） | **自己** | 概率视角分析、方法对比、理论延伸 |
| Idea 笔记 | `/idea-capture` | 理论动机、可行性评估、下一步方向，保存到 `Inspiration/` |

### 重要约定

汇总索引和主题概览**移动到 `Conclusion/` 目录**。

参考案例：2026-01-27 整理 `科研/CV/Generate/Paper/` 的 17 篇论文 → 3 主题概览 + 4 Idea → 汇总移至 `Conclusion/`

## Paper Frontmatter 规范

整理论文目录时，每篇论文笔记需要统一的 frontmatter：

```yaml
---
title: 论文完整标题（与文件名一致）
tags:
  - CV
  - Diffusion
  - CVPR
  - train-free
category: 图像编辑/图像恢复/逆问题/采样加速/Flow模型
created: YYYY-MM-DD
---
```

## Notes

- **搜索路径约束**：搜索限制在目标目录（如 `科研/CV`），避免全库搜索。参见 CLAUDE.md 的搜索路径约束
- **不删除用户原始数据**：保留操作记录（`merged_from` 字段）
- **不破坏现有内容**：整理时保留原始内容，只补充缺失部分
- **可逆操作优先**：执行前建议用户备份
